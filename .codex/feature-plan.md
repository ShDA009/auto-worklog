# Feature Requirements: Daily Tempo Worklog Automation

## Goal
Реализовать CLI-приложение на Go для ручного списания времени в Jira Tempo за выбранный день на основе:
- встреч из Outlook Calendar (Microsoft Graph);
- активности по Jira-задачам в рабочих статусах.

## Business Context
- Ручное списание времени занимает много времени и приводит к пропускам/неточностям.
- Нужен повторяемый ежедневный процесс, который учитывает календарные встречи и реальную работу по задачам.
- Пользователь запускает инструмент на рабочем ПК; VPN уже подключен; доступ к корпоративному Outlook и Jira есть.

## Functional Requirements
1. Инструмент запускается только вручную из CLI.
2. Поддерживаются команды:
- `worklog plan --date YYYY-MM-DD` (только расчет и вывод, без записи);
- `worklog apply --date YYYY-MM-DD` (запись в Tempo после подтверждения).
3. Для выбранной даты приложение получает встречи из Outlook через Microsoft Graph в таймзоне `Europe/Moscow`.
4. Для каждой встречи рассчитывается длительность в минутах и увеличивается на 20%.
5. Если заголовок встречи содержит ключ `ODP-XXXX`, время встречи списывается в эту задачу.
6. Если ключ `ODP-XXXX` не найден, время встречи списывается в задачу `ODP-2933`.
7. Приложение получает из Jira активность за день и строит интервалы по задачам в статусах:
- `In Progress`;
- `Подтверждение`.
8. Если задача в течение дня была переназначена на другого сотрудника, учитывается только интервал до момента переназначения.
9. После расчета встреч вычисляется остаток рабочего дня:
- `remaining_minutes = max(0, 480 - meetings_minutes_with_buffer)`.
10. Остаток распределяется по задачам в работе:
- при одновременной работе по нескольким задачам в один и тот же интервал времени минуты делятся поровну;
- агрегирование выполняется по issue key.
11. Если по Jira не найдено задач в работе, весь остаток списывается в `ODP-2933`.
12. Перед записью в Tempo выводится детальный предпросмотр:
- issue key;
- источник (`meeting` или `activity`);
- минуты;
- комментарий для worklog.
13. Команда `apply` выполняет запись только после явного подтверждения пользователя.
14. Повторный `apply` за ту же дату не создает дубли (идемпотентность).

## Non-Functional Requirements
1. Язык реализации: Go.
2. Конфигурация через переменные окружения:
- `MS_TENANT_ID`, `MS_CLIENT_ID`;
- `JIRA_BASE_URL`, `JIRA_EMAIL`, `JIRA_API_TOKEN`;
- `TEMPO_API_TOKEN`;
- `DEFAULT_ISSUE=ODP-2933`, `WORKDAY_HOURS=8`, `WORK_STATUSES=In Progress,Подтверждение`.
3. Секреты не хранятся в репозитории; токены сохраняются только локально (token cache).
4. Логи должны быть достаточными для диагностики, без вывода секретов.
5. Расчеты выполняются в минутах с детерминированным округлением.
6. Ошибки внешних API должны завершать команду с ненулевым кодом и понятным сообщением.

## Acceptance Criteria
1. `worklog plan --date <date>` выводит полный план списаний и не создает worklog в Tempo.
2. Встречи с `ODP-XXXX` списываются в соответствующие задачи.
3. Встречи без `ODP-XXXX` списываются в `ODP-2933`.
4. Для всех встреч применяется множитель `1.2` к длительности.
5. Остаток дня до `8h` корректно распределяется между задачами в статусах `In Progress` и `Подтверждение`.
6. При одновременной активности по нескольким задачам распределение выполняется поровну по интервалам пересечения.
7. При переназначении задачи учитывается только часть интервала до смены исполнителя.
8. `worklog apply --date <date>` требует явного подтверждения перед записью.
9. Повторный `worklog apply --date <date>` не создает дублирующие записи.
10. При недоступности Graph, Jira или Tempo команда завершается с ошибкой и диагностируемым сообщением.

## Edge Cases
1. За день нет встреч и нет задач в работе.
2. За день есть только встречи без `ODP-XXXX`.
3. Встречи пересекаются по времени.
4. Встреча отменена или имеет нулевую длительность.
5. Сумма встреч с буфером превышает 480 минут.
6. Задача несколько раз входит и выходит из рабочих статусов в течение дня.
7. Переназначение задачи происходит внутри рабочего интервала.
8. Временная недоступность Graph/Jira/Tempo.
9. Частично успешная запись при `apply` и повторный запуск.
10. Невалидный формат даты в CLI.

## Out of Scope (v1)
1. Автоматический запуск по расписанию.
2. Desktop/UI интерфейс.
3. Поддержка шаблонов ключей, отличных от `ODP-XXXX`.
4. Кастомные правила распределения времени по проектам/командам.
